# AI 辅助开发通用工作流

## 核心原则

1. **最小化起步** — 任何任务都从最小可运行单元开始，迭代推进
2. **信息先行** — 动手前充分收集信息，避免盲目编码
3. **渐进式交付** — 每次只完成一个原子步骤，验证通过后再继续
4. **持续验证** — 每个步骤完成后立即验证，及早发现问题
5. **关注耦合** — 任何模块变更前先理解其依赖关系

---

## 一、需求分析与分解（Requirement Decomposition）

### 1.1 原子化拆分

收到需求后，按以下层次逐级拆解：

```
需求 → 功能模块 → 原子任务 → 执行步骤
```

**拆分原则**：
- 每个原子任务应在 1 轮对话内可完成
- 每个原子任务应有可验证的产出
- 任务间依赖关系需显式标注

### 1.2 依赖图谱

绘制依赖关系图，明确：
- 哪些模块可以并行开发（无依赖或依赖已满足）
- 哪些模块必须串行（存在强依赖）
- 哪些是公共基础组件（应优先实现）

### 1.3 任务优先级排序

```
P0: 阻塞性基础（数据结构、核心接口、类型定义）
P1: 功能性核心（主要业务逻辑）
P2: 功能性次要（边界情况、错误处理）
P3: 增强性（性能优化、文档、完善测试）
```

### 1.4 需求澄清清单

开始编码前，先与需求方确认以下问题：

- [ ] 需求的核心目标是什么？（一句话描述）
- [ ] 成功的验收标准是什么？
- [ ] 是否存在已有实现或参考项目？
- [ ] 技术栈是否有强制约束？
- [ ] 性能/安全/合规是否有硬性要求？
- [ ] 部署环境是什么？
- [ ] 预期的用户规模和使用模式？

## 二、信息收集（Information Gathering）

### 2.1 信息分层收集策略

按优先级执行信息收集：

```
Layer 1: 项目上下文
  - 代码库现有架构、风格、约定
  - 已有配置文件（package.json, Cargo.toml, pyproject.toml 等）
  - 现有测试结构和覆盖率

Layer 2: 技术选型调研
  - 当前语言/框架的最新稳定版本
  - 官方文档推荐的最佳实践
  - 社区主流方案对比（使用 WebFetch）

Layer 3: 参考实现
  - 类似功能的开源实现参考
  - 文档中的示例代码
  - API / SDK 的使用模式

Layer 4: 约束条件确认
  - 环境依赖（Node/Python/Java 版本等）
  - 外部服务依赖（数据库 / 消息队列 / 第三方 API）
  - 团队技术栈惯性（已有技术沉淀）
```

### 2.2 信息收集手段

| 手段 | 适用场景 | 工具/方法 |
|------|----------|----------|
| 代码搜索 | 理解现有实现 | Grep、Glob、Read |
| 依赖分析 | 理解项目结构 | 检查 lock 文件、模块图 |
| 官方文档 | 理解 API/SDK | WebFetch |
| 社区资源 | 最佳实践方案 | WebFetch 搜索 |
| 现有测试 | 理解预期行为 | 阅读 test 目录 |

### 2.3 信息记录要求

在开始实现之前，必须输出以下任一：
- 方案设计文档（复杂任务）
- 实施清单（中等任务）
- 关键决策记录（简单任务）

## 三、实施流程（Implementation Process）

### 3.1 标准开发循环

```
[分析] → [收集信息] → [计划] → [实现] → [验证] → [提交]
                                ↑                    |
                                └─── 失败回滚 ────────┘
```

1. **分析**: 理解需求，拆分为原子任务
2. **收集信息**: 收集所需的技术信息
3. **计划**: 确定实现步骤和优先级
4. **实现**: 编写代码（坚持最小变更原则）
5. **验证**: 运行测试、lint、类型检查
6. **提交**: 仅当验证全部通过

循环周期：**1 次循环 = 1 个原子任务**

### 3.2 多文件变更原则

- 一次只变更一个逻辑单元
- 如果涉及多个文件，优先修改依赖最少的文件
- 公共接口变更前，先确认所有调用方
- 重构和功能新增分开提交

### 3.3 粒度控制

```
❌ 错误示例：一次提交修改了 3 个模块、2 个测试文件、重构了 1 个接口
✅ 正确示例：一次提交只修改 1 个模块，附带其测试文件
```

## 四、决策框架（Decision Framework）

遇到需要决策的问题时，按以下步骤执行：

### 4.1 技术选型决策树

```
1. 项目是否有既定的技术栈？
   ├── 是 → 遵循现有技术栈
   └── 否 → 2. 需求的核心诉求是什么？
            ├── 性能优先 → 选择编译型/静态语言方案
            ├── 开发效率优先 → 选择解释型/动态语言方案
            ├── 生态丰富 → 选择社区活跃的方案
            └── 长期维护 → 选择标准化程度高的方案
```

### 4.2 方案选择矩阵

| 维度 | 问题 |
|------|------|
| 必要性 | 这个方案解决的核心问题是什么？ |
| 简洁性 | 是否有更简单的替代方案？ |
| 兼容性 | 是否与现有系统兼容？ |
| 可维护性 | 6 个月后是否还能理解？ |
| 性能 | 是否满足预期性能指标？ |

### 4.3 默认拒绝清单

除非有充分理由，否则默认避免：
- 引入新依赖（但只用了一小部分功能）
- 使用实验性/非稳定特性
- 过度设计（YAGNI）
- 在现有项目中引入新语言/新框架

## 五、质量保障（Quality Assurance）

### 5.1 验证流水线

每个原子任务完成后，必须执行：

```
1. [编译/解释检查]  代码能否正确构建/运行？
2. [Lint 检查]      是否符合代码规范？
3. [类型检查]       类型注解是否正确？
4. [单元测试]       新增功能是否有测试覆盖？
5. [集成测试]       变更是否破坏了现有功能？
6. [边界测试]       边界情况和错误处理是否完善？
```

### 5.2 代码质量自检

提交代码前自问：

- [ ] 这个变更是否做到了最小（只做必要的事）？
- [ ] 命名是否清晰表达意图？
- [ ] 是否有重复代码可以抽象？
- [ ] 错误处理是否完整？
- [ ] 边界情况是否覆盖？
- [ ] 是否有调试代码遗留（print/console.log/breakpoint）？
- [ ] 是否需要更新文档/注释？

### 5.3 失败处理

- 验证失败时，**回滚变更**至上一个已知工作状态
- 重新分析失败原因，调整方案后再试
- 同一任务连续失败 2 次，应暂停并重新评估方案

## 六、沟通与报告（Communication）

### 6.1 进度报告格式

完成任务后，按以下格式汇总：

```
## 完成: <任务名>
- 变更文件: file1.py, file2.py
- 新增功能: 简短描述
- 验证结果: lint ✅ | test ✅ | type ✅
- 遗留事项: 下次迭代需处理的问题
```

### 6.2 遇到阻塞

遇到阻塞时立即报告：
- 阻塞的具体问题是什么？
- 已经尝试了哪些方案？
- 建议的解决方案或备选路径？

## 七、跨技能协作（Cross-Skill Integration）

本工作流与其他 skills 的关系：

| Skill | 触发时机 | 协作方式 |
|-------|----------|----------|
| `project-scaffolding` | 新项目创建或需要理解项目结构时 | 加载后自动检测项目类型并适配布局 |
| `testing-strategy` | 需要编写或运行测试时 | 加载后自动检测已有测试框架并适配 |
| `python-style` | Python 项目编码时 | 加载 python-style 确保代码风格 |

**协作方式**：所有 skill 都遵循"检测-适配"模式而非"预设-执行"模式，
因此即使项目使用非常规技术栈，agent 也会先探测再行动。
